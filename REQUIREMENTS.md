# Requirements Analysis — Shared Expenses App

> Status: Draft v1 · Owner: engineer of record · Assignment: Spreetail "Build a Shared Expenses App"

## 1. Problem statement

A group of flatmates with **changing membership**, **mixed currencies** (INR + USD), and a
**messy historical spreadsheet** needs an app that imports that spreadsheet *safely*, tracks
who-owes-whom over time, and lets them settle up — where every number is auditable and every
data fix is approved by a human before it touches the ledger.

## 3. Functional requirements

### Authentication
- **R-AUTH-1** Register and log in.
- **R-AUTH-2** Only authenticated users may access groups/expenses.

### Groups & membership
- **R-GRP-1** Create/rename a group; a group has a base currency (INR).
- **R-MEM-1** A **person** in a group is either a **member** or a **guest** (Dev, Kabir are guests).
- **R-MEM-2** Membership is time-bounded: `join_date`, optional `leave_date`.
  Meera: Feb 1 – Mar 31. Sam: Apr 8 – present.
- **R-MEM-3** The system can answer "who was a member on date X" — this gates split validation.

### Expenses & splits
- **R-EXP-1** Create / edit / void an expense (payer, amount, currency, date, description, split).
- **R-EXP-2** Support all four split types present in the data: `equal`, `unequal` (absolute),
  `percentage`, `share` (weights).
- **R-EXP-3** A split's participants may be a subset of members (a member can be legitimately
  excluded) and may include guests.
- **R-EXP-4** Persist original amount + currency **and** base-currency amount + FX rate used.
  Never destroy the source value.
- **R-EXP-5** Voided / superseded expenses are retained for audit, never hard-deleted.

### Balances & settlement
- **R-BAL-1** Net balance per person = paid − owed + sent − received (money you sent pays down
  your debt; money you received increases it). **Derived, never stored.**
- **R-BAL-2** Simplified view: minimal who-pays-whom list (greedy min-cash-flow, heuristic).
- **R-BAL-3** Itemized view: every split row contributing to a person's balance.
- **R-SET-1** A settlement/payment (from → to, amount, date) is its own entity, not an expense.

### Import & anomaly pipeline (graded core)
- **R-IMP-1** Ingest `expenses_export.csv` as-is — no manual pre-editing, no crash on any row.
- **R-IMP-2** Detect every anomaly and surface it in an Import Report (row #, type, detail, action).
- **R-IMP-3** Handle each anomaly by a documented policy; nothing silently guessed or dropped.
- **R-IMP-4** Approval gate: detected mutations (merge, void, reclassify, exclude, correct) sit in
  quarantine until a human approves / rejects / edits.
- **R-IMP-5** Commit writes only approved rows to the ledger; rejected/pending rows are held, not lost.

### Reporting
- **R-REP-1** The Import Report is generated by the app, not written by hand.

## 4. Non-functional requirements

- **R-NFR-1** Money is exact: integer minor units (paise/cents) for division; remainder distributed
  deterministically; totals reconcile to the cent.
- **R-NFR-2** Relational DB only (PostgreSQL).
- **R-NFR-3** Auditability: every canonical row traces to its import batch + source row; every
  anomaly action traces to who approved it.
- **R-NFR-4** Deployed and reachable at a public URL.
- **R-NFR-5** Meaningful commit history (per-feature / per-detector commits).
- **R-NFR-6** Explainability: any anomaly can be traced through the code in a live session.

## 5. Scope

**In scope:** everything above.

**Out of scope (deliberate — recorded so it reads as a decision, not a miss):**
- Live FX API (a documented date-effective rate table is used instead).
- Multi-group dashboards, notifications, mobile app, receipt uploads.
- Provably-optimal debt minimization (greedy heuristic; the problem is NP-hard).
- Auto-matching partial settlements to specific expenses.

## 6. Definition of done

A feature is done when it works end-to-end in the deployed app, has a committed test or manual
trace, and — if it touches data policy — is documented in SCOPE.md / DECISIONS.md.

Submission checklist: deployed URL · commit history · README.md · SCOPE.md · DECISIONS.md ·
AI_USAGE.md · generated Import Report.

## 7. Open decisions — proposed resolutions (pending confirmation → move to DECISIONS.md)

| # | Decision | Proposed resolution | Rationale |
|---|---|---|---|
| D1 | Duplicate tie-break (Thalassa 2400 vs 2450) | Quarantine + user picks; no auto-merge | Splitwise surfaces, never silently merges; satisfies Meera's approval gate |
| D2 | Percentage ≠ 100 (Pizza 110%) | Hard-fail, require correction to 100% | No mainstream app saves a non-balancing split |
| D3 | Ambiguous date (May 4 vs Apr 5) | Always prompt; never default | Date choice changes whether Sam is in the split — materially affects a balance |
| D4 | FX rate basis | Date-effective rate table, documented manual rates; store original + converted + rate | Splitwise uses user-set rates, not live conversion; auditable |
| D5 | Rounding remainder | Deterministic: integer paise, remainder to first R by fixed order (id/alpha) | Reproducible + testable + explainable; beats Splitwise's random for audit (Rohan) |
| D6 | Backend stack | Lean Django + DRF (JD-aligned, batteries for 2-day build); FastAPI acceptable if preferred | Own the choice in DECISIONS.md either way |
